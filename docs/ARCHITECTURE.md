# Formless — Architecture

> Version 1.0 — 2026-03-12

---

## Overview

Formless is a web application with three distinct surfaces:

1. **Admin Portal** — schema builder, record management, integrations
2. **Intake Surface** — the end-user facing input (web, Slack, email)
3. **AI Engine** — the classification, extraction, and validation pipeline

---

## Tech Stack

### Frontend
- **Next.js 14** (App Router) — unified frontend + API in one repo
- **TypeScript** — strict mode throughout
- **Tailwind CSS** — utility-first styling
- **shadcn/ui** — component primitives (same as Spendable/ToolForge)

### AI
- **Claude API** (`claude-sonnet-4-6`) — primary model for classification + extraction
- **Prompt architecture:** two-pass pipeline (classify first, then extract)
- **Structured output:** JSON mode / tool use for guaranteed schema conformance
- **Zod** — runtime validation of AI output (proven in nlp-cms prototype)

### Backend
- **Next.js API Routes** — MVP simplicity, monolith-first
- **Zod** — schema validation layer (server-side)
- **Supabase** — PostgreSQL + auth + real-time (fast to ship)

### Database Schema (High Level)

```
workspaces
  id, name, slug, created_at

users
  id, workspace_id, email, role (admin|member), created_at

schemas
  id, workspace_id, name, description, definition (jsonb), version, created_at

records
  id, schema_id, workspace_id, raw_input (text), extracted_data (jsonb),
  schema_version, status (open|in_progress|resolved), created_at, updated_at

record_edits
  id, record_id, user_id, field_name, old_value, new_value, created_at

intake_sessions
  id, schema_id, session_token, messages (jsonb), status, created_at
```

### Auth
- **Supabase Auth** (email/password + Google OAuth) for MVP
- Upgrade path to Clerk if SSO requirements emerge

### Deployment
- **Vercel** — Next.js native, zero-config
- **Supabase** — managed Postgres, no infra to maintain

## AI Pipeline Architecture

### Pass 1: Classification
```
Input: raw text from user
Prompt: "Given this schema definition and this text, does the text contain
         an intake request for this schema? Respond yes/no with confidence."
Output: boolean + confidence score
```

For MVP with single-schema intake URLs, classification is implicit (the URL determines the schema). Classification becomes important for multi-schema workspaces and Slack/email channels.

### Pass 2: Extraction
```
Input: raw text + schema definition (field names, types, required flags, enum values)
Prompt: "Extract the following fields from this text. Return ONLY valid JSON
         conforming to this schema. For missing optional fields use null.
         For missing required fields use the string '__MISSING__'."
Output: JSON object
```

### Pass 3: Validation
```
Input: extracted JSON
Process: Zod parse against generated schema
Output: validated record OR list of missing/invalid fields
```

### Pass 4: Clarification (if validation fails)
```
Input: list of missing required fields + conversation history
Prompt: "The user submitted a request but the following required fields
         could not be extracted: [fields]. Generate ONE natural follow-up
         question to get this information."
Output: natural language question → shown to user
```

## Schema Definition Format

Formless stores schemas as a structured JSON definition. Example for IT Tickets:

```json
{
  "name": "IT Support Ticket",
  "fields": [
    { "name": "title", "type": "string", "required": true, "description": "Brief summary of the issue" },
    { "name": "description", "type": "string", "required": true, "description": "Full details of the problem" },
    { "name": "category", "type": "enum", "required": true, "values": ["hardware", "software", "network", "access", "other"] },
    { "name": "priority", "type": "enum", "required": true, "values": ["low", "medium", "high", "critical"] },
    { "name": "requester", "type": "string", "required": true },
    { "name": "affected_system", "type": "string", "required": false }
  ]
}
```

This definition is generated by AI from the admin's natural language description, then validated and stored. It's also the input to the extraction prompt at intake time.

## Monorepo vs. Single App

**Decision: Single Next.js app for MVP.**

Rationale:
- Admin portal and intake surface share components, types, and AI service
- Supabase handles auth and DB — no separate backend service needed
- Can split into separate services in Phase 2 if load demands it

---

## Key Design Decisions

| Decision | Choice | Reason |
|----------|--------|--------|
| AI model | Claude claude-sonnet-4-6 | Best structured output quality, already familiar |
| Two-pass AI | Classify then extract | Reduces hallucination, cheaper than one complex prompt |
| Zod validation | Required | Proven in prototype, catches AI errors before they hit DB |
| Supabase | PostgreSQL + Auth in one | Fastest path to working auth + DB |
| Next.js monolith | Yes for MVP | Avoid infra complexity while proving the product |
| JSONB for records | Yes | Schema evolves; JSONB lets us store any shape without migrations |

---

## Security Considerations

- Intake URLs are unique tokens (not sequential IDs) — hard to enumerate
- AI output is always validated before storage — no raw AI output in DB
- Workspace isolation enforced at query level (all queries include `workspace_id`)
- API keys (Claude) stored as environment variables, never client-side
- Rate limiting on intake endpoints to prevent abuse
